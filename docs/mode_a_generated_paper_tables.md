# Mode A: Generated Paper Table Script Harness

Stage 3R.4.1 adds a controlled execution harness for selected legacy paper table scripts.

The goal is to run the selected scripts inside a temporary compatibility workspace and capture:

- script status,
- stdout/stderr logs,
- generated or modified files,
- copied table outputs,
- validation status.

## Run

First make sure archive replay and generated summaries are available:

    python scripts/trace.py mode-a --audit --clean --generated-summaries

Then run the paper table harness:

    python scripts/53_run_paper_table_scripts.py --clean
    python scripts/54_validate_paper_table_outputs.py

Or use the unified Mode A entry after the option is enabled:

    python scripts/trace.py mode-a --audit --clean --generated-summaries --paper-tables

## Outputs

Generated outputs are placed under:

    analysis/paper_generated/paper_tables/

Important files:

- `paper_table_script_run_manifest.json`
- `paper_table_script_run_report.md`
- `paper_table_validation_report.json`
- `paper_table_validation_report.md`
- `logs/*.stdout.txt`
- `logs/*.stderr.txt`
- `script_outputs/**`

## Scope

This step validates the execution harness for legacy paper table scripts.

It does not yet claim byte-identical reproduction of every paper table.

## Availability-aware validation

The Stage 3R.4.1 validator checks both files newly generated by legacy scripts and paper-exact files available in the final compatibility workspace.

This is necessary because some legacy scripts use pre-existing paper-exact intermediate tables without rewriting them when they already exist.

A PASS means:

- the selected analysis/table scripts executed successfully;
- required paper-table output groups are available after the run;
- stdout/stderr logs and output manifests were captured.

It does not yet mean byte-identical reproduction of every paper table. That is handled by the next equivalence-validation step.
